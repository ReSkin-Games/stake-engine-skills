## Overview

All game outcomes are **pre-generated by the developer**. The RGS does not calculate wins in real time — it selects a result from data files the developer uploads. The frontend is purely an animation layer over the result the RGS returns.

This is why every game must include a disclaimer making clear that "winnings are settled according to the amount received from the RGS and not from events within the web browser".

## What the Developer Provides

For each betmode, two file types are submitted:

### Lookup Table (LUT)

A `.csv` file (no header row) with three columns:

| Column | Description |
|--------|-------------|
| `sim_id` | Simulation number — matches the line index in the book. |
| `weight` | Selection weight — proportional probability of this outcome being chosen. |
| `payoutMultiplier` | Payout multiplier as an integer in cents (×100). |

```csv
0,100,0
1,100,0
2,50,150
3,10,2500
```

In this example, `sim_id=3` has a payout of 25.00x (2500 / 100) and a weight of 10.

The minimum multiplier granularity is `1` (= `0.01x`). Two decimal places only — values like `0.001x` or `0.015x` are not possible.

### Book

A `.jsonl` file compressed with **ZStandard** (`.jsonl.zst`). Each line is one simulation — the full set of events the frontend needs to animate the round.

Required fields per line:

| Field | Description |
|-------|-------------|
| `id` | Line number (0-indexed). Must match `sim_id` from the LUT. |
| `payoutMultiplier` | Must match the LUT value for the same `sim_id`. |
| `events` | Array of game events the frontend renders. |

```json
{"id":0,"payoutMultiplier":0,"events":[{"type":"spin","symbols":[[1,2,3],[4,5,6],[7,8,9]]}]}
{"id":2,"payoutMultiplier":150,"events":[{"type":"spin","symbols":[[1,1,1],[4,5,6],[7,8,9]]}]}
```

Mismatches between `id` and line index, or between book `payoutMultiplier` and LUT `payoutMultiplier`, cause validation errors at upload.

## Outcome Selection Algorithm

The RGS uses **weighted random selection** over the LUT:

1. **Build cumulative weights** — the RGS sums all weights, assigning each `sim_id` a numeric range proportional to its weight (e.g. loss 0–100, small win 100–150).
2. **Generate a random number** — between 0 and the total weight sum.
3. **Binary search** — find which range contains the number. O(log N) and mathematically precise.
4. **Load the simulation** — the matched `sim_id` is used to read the corresponding JSON line from the book, which is returned to the player.

Larger weight → larger range → higher selection probability. RTP is determined entirely by the relationship between weights and payout multipliers in the LUT.

## Stateless Model

The RGS does not store history. Every round is independent:

- Player actions cannot influence the outcome — selection is pure RNG over LUT weights.
- Every round has the same odds, regardless of previous results.
- Only outcomes defined in the LUT and book can occur — there is no synthesis of novel results at runtime.

This protects RTP determinism and prevents manipulation.

## Betmodes

Betmodes are distinct game modes with **independent math models** — each has its own LUT and book. They can have entirely different RTP, volatility, and win distributions. Common examples: `BASE`, `SUPER` (bonus buy), turbo variants. The frontend selects the betmode in the `/wallet/play` request via the `mode` field.

## Wincap

Wincap is the maximum payout a player can win in a single round:

- Set **per betmode** — different modes can have different caps.
- Defined by the publisher when creating the math model.
- Wins exceeding the cap are automatically capped by the RGS.
- Controls volatility and prevents runaway single-round payouts.

## Session Lifecycle

1. The operator platform launches the game with a `sessionID` and `rgs_url` in the URL.
2. The frontend calls `POST /wallet/authenticate` with the `sessionID`. This must be the first call — other endpoints return `ERR_IS` until authenticated.
3. Authenticate returns balance, game config (bet levels, etc.), and any active round.
4. Subsequent calls (`/play`, `/end-round`, `/balance`, `/bet/event`) reuse the same `sessionID`.
5. Sessions have a limited lifetime. If a session expires, the player must relaunch the game to receive a new one.
6. Re-calling `/wallet/authenticate` with the same `sessionID` is safe and idempotent.

## Round Lifecycle

1. **Play** — `POST /wallet/play` with `{sessionID, amount, mode}`. The RGS debits the bet, selects an outcome via weighted random selection, and returns `{balance, round}` with `payoutMultiplier`, `costMultiplier`, and `state`.
2. **In-round events** — the frontend may call `POST /bet/event` to record player actions during the round; these enable mid-round state recovery on reconnect.
3. **End** — the round-end behavior depends on payout:
   - `payoutMultiplier == 0` — the RGS **auto-completes** the round. No `end-round` call is required.
   - `payoutMultiplier > 0` — the frontend **must** call `POST /wallet/end-round` to credit winnings to the player.
4. **Resume** — if the frontend reloads mid-round, `POST /wallet/authenticate` returns the active round in the `round` field. The frontend should display the outcome and then call `/wallet/end-round` (if `payoutMultiplier > 0`).
5. **`round: null` after recent play** — means the round had a zero payout and was auto-completed. Expected behavior.

## Bet Replay

Replay is a public, session-less mechanism: `GET /bet/replay/{game}/{version}/{mode}/{event}` returns the same `{payoutMultiplier, costMultiplier, state}` as a `/wallet/play` response. Replay URLs can be shared in chat, social media, or used for support investigations. The frontend must support a `replay=true` mode that fetches replay data, disables all bet controls, plays back the animation, and offers a Play Again button. See `rgs-api.md` for the full endpoint contract.
